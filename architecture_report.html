Important Note on Industry Choice

Some industries look like placeholders

The "Hospitality" industry uses Yelp as the focal company and compares it to TripAdvisor and OpenTable — these are review platforms, not hospitality businesses. The "Restaurants" industry uses McDonald's. These feel like they were seeded quickly to demo the platform rather than reflect a real use case. Worth revisiting what industries you actually want to analyze.

Architecture

How the system is structured end-to-end.

Three-layer system

A React frontend talks to a FastAPI backend via REST + SSE. The backend runs a Python pipeline that fetches reviews, scores them with an LLM, and caches results in SQLite. Everything has a mock fallback.

Data Flow

Step-by-step of what happens when you load the dashboard or trigger an analysis.

On Every Page Load (no analysis triggered)

1

Frontend requests GET /api/industry/{industry}

Happens automatically when the dashboard loads for the selected industry.

2

Backend checks its in-memory cache

If this industry has been requested before, returns cached result immediately.

3

If not cached: loads mock scores from industry_config.py

Hardcoded scores like HubSpot: {ease_of_use: 78, feature_richness: 72...}

4

Tries to load review counts from local JSON file

e.g. data/sample_reviews.json — counts how many reviews per company. If found, isMock = false. If not found, isMock = true.

5

Synthetic data is always generated here

Sentiment trends, churn flows, praise/complaint themes — all procedurally built from the scores. Never from real data.

6

Frontend renders dashboard

If isMock=true: shows amber "Demo data" banner. All charts render with mock scores.

When "Run Analysis" is triggered

1

Frontend posts to POST /api/run-analysis/stream

Opens an SSE connection to stream progress events back to the UI.

2

Stage: FETCH

Tries live sources (Apify / Kaggle) if use_live_sources=true. Falls back to local JSON file. If nothing found → reviews = null.

3a

❌ No reviews found → Mock fallback

Returns mock_scores + MOCK_INSIGHTS_BY_INDUSTRY. Sets data_source = "mock_no_reviews". Done.

3b

❌ No GROQ_API_KEY → Mock scores fallback

Reviews found, but no LLM client. Returns mock scores + mock insights. Sets data_source = "mock_scores_local_file". Done.

3c

✅ Reviews found + API key exists → Real LLM analysis

Continues to real scoring.

4

Stage: SCORING (real)

Each review scored 0-100 on 6 dimensions by llama-3.1-8b-instant. Parallelized with 8 workers. Up to 64 reviews sampled.

5

Stage: INSIGHTS (real)

Aggregated scores sent to llama-3.3-70b-versatile. Returns 8-12 strategic insights. Sets data_source = "live_pipeline_local_file".

6

Stage: ENRICHING (real market data)

Parallel fetch of optional enrichments — each silently skipped if its key is absent:
• Google News (SER_API) → headlines for focal company → newsHeadlines
• Alpha Vantage (Alpha_vantage) → live stock quotes → financeData
• Glassdoor via SerpAPI (SER_API) → employee ratings → glassdoorData
• Google Trends (SER_API) → sentiment timeline + share-of-voice
These fields are audience-gated in the frontend and only appear for the relevant view.

7

Cache updated, isMock = false

Dashboard reloads with real scores, insights, and live market data cards. Amber banner disappears.

Real vs Fake Data

Every single piece of data in the dashboard — where it comes from and whether it can be real.

Still Synthetic (no real-data equivalent yet)

Churn / Migration Flows

Generated by industry_service.py using hash(f"{source}{target}") % 15 + 5. No public API tracks actual customer migration between products. Could be approximated with SerpAPI Google Trends comparison queries in a future iteration.

Conditionally Real (depends on which APIs are configured)

Sentiment Trends Chart — 3-tier priority

Tier 1 (best): Computed from real review ratings grouped by date — averaged per quarter per company. Active when Apify/Kaggle/SerpAPI Maps fetches return dated reviews.
Tier 2: SerpAPI Google Trends — real historical search-interest timeline (0-100 normalised to -1/+1). Active when SER_API is set and Tier 1 not available.
Tier 3 (fallback): _synthetic_sentiment() procedural formula — only used when both tiers above fail.

Praise / Complaint Themes — 2-tier priority

Tier 1 (real): LLM (Groq) analyses actual review text and returns top 3 praise themes + top 3 complaint themes per company, plus 0-100 scores. Active after "Run Analysis" with GROQ_API_KEY.
Tier 2 (fallback): Score-sum formula — dimension scores ≥ 60 counted as praise, < 60 as complaint.

Share of Voice — 3-tier priority

Tier 1: Real review counts from Apify/Kaggle/SerpAPI Maps fetches.
Tier 2: Real review counts from local JSON sample files.
Tier 3: SerpAPI Google Trends relative search interest (proportional, not absolute counts).
Last resort: Hash-based fake numbers (only when all tiers fail).

Dimension Scores (Radar, Heatmap, Dimension Deltas)

Without API key: Hardcoded from industry_config.py mock_scores.
With GROQ_API_KEY + Run Analysis: LLM scores each review and aggregates. Real numbers from actual review text.

Insights Panel

Without API key: 5 pre-written insights per industry from MOCK_INSIGHTS_BY_INDUSTRY.
With GROQ_API_KEY + Run Analysis: LLM generates 8-12 fresh insights based on actual scored data.

Always Real (keys set, files exist)

Google Maps Reviews — Restaurants, Hospitality, Travel

resources/serpapi_adapter.py fetches real Google Maps customer reviews for location-based industries. Active when SER_API is set. Reviews feed directly into LLM scoring pipeline.

Review Counts (when files exist on disk)

All 12 data/sample_reviews_*.json files exist. Review counts are loaded from these and are real counts of real sample reviews.

isMock Flag

Correctly set in industry_service.py as not bool(review_counts). The frontend amber banner shows accurately.

Current State of This App

You are in Scenario C/F. GROQ_API_KEY, APIFY_API_TOKEN, and SER_API are all set in .env. What's Missing

Gaps by priority — updated after SerpAPI + real data integration.

High Priority

No database migrations

SQLAlchemy create_all() is used. Any schema change will silently fail on existing databases. Alembic is not set up.

Sentiment Trends data source label missing in UI

The chart now shows real data (Google Trends or dated-review aggregation) but the UI doesn't tell the user which source was used. A small badge ("Source: Google Trends" vs "Source: Review ratings") would add trust.

Churn / Migration Flows still hash-based

No public API tracks actual product switching. Options: (1) approximate with SerpAPI Trends comparison, (2) remove the chart from investor view, (3) label clearly as "illustrative".

Medium Priority

Personalization only runs once

Computed at onboarding completion. Dashboard chat doesn't refresh it. Users need to reset onboarding to regenerate.

Chat sessions accumulate forever

No TTL or cleanup job. SQLite will grow unbounded over time.

OpenWebNinja Glassdoor API not wired (key is set)

The Openweb_ninja key is in .env but the correct endpoint path on api.openwebninja.com is unknown. The current Glassdoor data comes from a SerpAPI Google Search scrape which is less reliable. Once the correct path is found, this is a drop-in improvement.

Hospitality industry has wrong focal company

Yelp is a review platform, not a hospitality business. Should be revisited.

SerpAPI Google Maps only covers 3 industries

Currently gated to restaurants, hospitality, travel. Could expand to any industry with a physical presence (banking branches, telecom stores, etc.).

Low Priority

No chart export (PNG/PDF)

CSV export exists but no visual export of charts.

No usage analytics

No tracking of which industries or charts are used most.

No bulk analysis

Can only run analysis on one industry at a time.

SQLite → Postgres needed before deployment

SQLite won't handle concurrent users or ephemeral cloud containers.

OpenWebNinja Glassdoor API not yet wired

Key Openweb_ninja is set in .env. The correct endpoint path is unknown — the provider's docs are behind login. Once found, implement resources/openwebninja_adapter.py and wire into fetch_glassdoor_for_industry() as the primary source, replacing the SerpAPI scrape fallback.

Alpha Vantage free tier rate limits

Free tier: 25 requests/day, 5/min. With up to 6 companies per industry, a single "Run Analysis" can consume 6 quota units. Consider caching finance data longer (it only changes on trading days) or batching requests.
